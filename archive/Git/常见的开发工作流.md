# 常见的开发工作流

工作流并不是一成不变的，在每个公司甚至每个团队的开发流程都可能不一样。不过各种的工作流规范都是为了提高项目流程管理的能力和开发协作的效率，尽量去避免一些可能的问题。

工作流没有最好的，只有最合适的，每种工作流都有它合适的一方面和不适用的一方面。在不同的项目、不同的阶段、不同的开发人数、不同的项目权限管理上都会出现各种不同的工作流形式。这里主要介绍我经历的几种工作流形式。



# 一、集中式工作流

最早基础版本管理软件是 SVN，当时是在上大学的时候，和 10 多个同学一起学习开发一个完整的项目，当时我们用 SVN 进行协作，所有的仓库代码都在同一个单点存储，所有的改动，大家都能近乎实时的知晓。

像 SVN 一样，集中式工作流以中央仓库作为项目所有修改的单点实体。同时大家都在同一个缺省的分支上进行开发。开发者开始工作时先克隆中央仓库，然后编辑文件和提交修改，修改完成后在合适的时间推送到中央仓库（SVN 的提交是直接推送的，GIT 的提交需要单独再 push）。 其他开发以一定的频率拉取下中央仓库的改动，保持自己的代码和中央仓库是同步的。
